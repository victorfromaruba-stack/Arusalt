# AruSalt website ‚Äî project memory

AruSalt is a small, real hand-harvested sea salt brand from Aruba. This project is a single-file HTML + Tailwind CDN website (no build step, no framework). Two products exist today, sold in-store only: Pure Caribbean Sea Salt ($29) and Caribbean Bath Salt ($49). We plan to add online ordering through Shopify.

## Content editing system (no-GitHub-access editors)

Some copy is editable without touching GitHub, via a small Decap CMS instance at `/admin`:
- `content/site.json` holds the editable fields (hero headline/subhead, announcement bar text, both products' description/price/net weight, the retailers list, newsletter blurb). Non-editable content stays hardcoded in `index.html` as before.
- A hydration script at the end of `index.html`'s `<script>` block fetches `content/site.json` on page load and patches it into elements tagged `id="cms-*"` (or `class="cms-marquee-line1/2"`), rebuilding `#cms-retailers-grid` entirely from the `retailers` array. If the fetch fails for any reason, the static HTML already in the page is the fallback ‚Äî the page never breaks from this.
- `admin/config.yml` defines the Decap CMS field schema (must stay in sync with `content/site.json`'s shape, and with the `id`s in `index.html` that the hydration script reads).
- `admin/index.html` is the CMS login/editor page. It has its own separate CSP meta tag (more permissive than the main site's, since it needs to load the Decap CMS bundle from unpkg and Netlify Identity) and is excluded from search indexing (`robots.txt` + `noindex` meta).
- Backend is `git-gateway` + Netlify Identity ‚Äî editors log in with an email/password Netlify Identity account (invited by the site owner), never a GitHub account. Netlify proxies the actual git commits using the owner's GitHub authorization, done once during setup.
- `publish_mode: editorial_workflow` ‚Äî edits sit as drafts/in-review before anything merges to `main` and goes live, rather than publishing instantly. Deliberate given the owner's security priorities; can be simplified to `publish_mode: simple` later if the review step proves like unnecessary friction.
- When adding a new editable field: add it to `content/site.json`, add the matching field block to `admin/config.yml`, tag the target element in `index.html` with a `cms-*` id/class, and add the corresponding lookup in the hydration script.
- The hydration script also keeps the Product JSON-LD (`<script type="application/ld+json" id="product-schema">`) price fields in sync with `content/site.json`'s prices ‚Äî parses the JSON, updates `offers.price` by matching product name against "Bath Salt", re-serializes. If a new CMS field should also be reflected in a JSON-LD block (schema.org data isn't otherwise touched by the hydration script), extend this same pattern rather than leaving structured data to silently drift from what an editor sets via `/admin`.

## Design system ‚Äî don't drift from this without flagging it to me

- **Fonts:** Space Grotesk for headings (h1‚Äìh4), Inter for body text
- **Colors:** ocean blue `#0ea5e9` (primary, used broadly), coral `#ef6a72` (secondary accent ‚Äî used sparingly, not as a default)
- **Signature motif:** a small rotated-square "diamond" shape echoing the crystal-lattice logo mark, reused as section dividers. A second, intentional divider shape ‚Äî a wave, echoing the logo's wave motif ‚Äî is used specifically at two full-section seams (`.wave-divider`, before "Respect for land and sea" and before "History"); this coexists with the diamond by design (confirmed with the site owner), not an inconsistency to converge.
- **Icons:** hand-drawn inline SVG (no image files), thin-stroke (~1.3 stroke-width on a 24√ó24 viewBox), plain dark (`text-gray-900`) ‚Äî matching the real Shopify store's icon style, not the site's blue/coral accents. Exception, confirmed with the site owner: functional/interactive UI icons (back-to-top arrow, FAQ accordion chevrons, sustainability checklist marks) use ocean-blue and a thicker 1.5‚Äì2px stroke instead ‚Äî color signals interactivity/status for these specifically, distinct from purely illustrative icons.
- **Section backgrounds:** alternate white and a pale ocean-tinted blue (`#f4f9fc`) ‚Äî never plain Tailwind gray. Non-`content-section` interstitial elements (e.g. the coral Gift Banner) are deliberately excluded from the alternation count; when inserting/removing a full section, recompute the white/pale sequence for everything downstream.

Products are in-store-only for now; the Shopify listings are deliberately kept in place (sold out) so online ordering can be switched on later without rebuilding the store. Keep "AVAILABLE EXCLUSIVELY IN STORES" / "Find a Retailer" copy as-is until priority #2 below actually goes live.
- Run `qa-validator` after any HTML/CSS/JS change, before telling me something is done.
- `index.html` has a Content-Security-Policy meta tag. If you add a new external resource (script, stylesheet, font, fetch target), update the CSP to allow it explicitly ‚Äî don't loosen it to a wildcard, and don't add `unsafe-eval` to the main site's CSP (only `admin/index.html`, a separate document, has that, because the Decap CMS bundle needs it).
- GitHub Pages can't serve custom HTTP headers, so `X-Frame-Options`, `X-Content-Type-Options`, and CSP's `frame-ancestors` directive can't be set from this repo at all ‚Äî that's a platform limitation, not something to keep re-flagging as an open bug.
- **Tailwind JIT trap, easy to reintroduce:** any custom/brand color that isn't a real Tailwind config token (`theme-blue`, `coral`, `coral-pale`, etc.) silently gets zero generated CSS for variant classes built on it ‚Äî `hover:`, `focus:`, `border-`, `shadow-{color}/{opacity}`, and similar ‚Äî because Tailwind's CDN JIT only generates what it recognizes from its own palette. This has bitten `hover:text-theme-blue`, `border-theme-blue`/`focus:border-theme-blue`, and `shadow-theme-blue/20` so far (all now fixed with explicit CSS rules in the `<style>` block). When adding a new utility-lookalike class on a custom color, add the backing CSS rule in the same change and verify with `getComputedStyle()` in a real browser ‚Äî don't trust that the class "looks like" a Tailwind utility. Note also that Tailwind's CDN-generated stylesheet is injected *after* this file's own `<style>` block, so for properties Tailwind sets via shared CSS custom properties (like `box-shadow`'s `--tw-shadow`/`--tw-shadow-colored` chain), a same-specificity override in `<style>` can silently lose the cascade ‚Äî verify computed values, and reach for a direct property override (optionally scoped by id, with `!important` if needed) rather than trying to replicate Tailwind's internal custom-property mechanism.
- The legal policy modal locks the background with `inert` on a `backgroundRegions` array (nav wrapper, mobile menu, WhatsApp widget, back-to-top button, skip-link, `<main>`, `<footer>`) whenever the modal opens. Any new element added as a direct `<body>`-level sibling (not inside `<main>`/`<footer>`/nav/mobile-menu) needs to be added to this array too, or it stays focusable/interactive while the modal is open ‚Äî this has been the single most common regression this project introduces when adding new floating/fixed UI. The mobile menu has its own equivalent (`menuBackgroundRegions`: `<main>`, `<footer>`, WhatsApp widget, back-to-top ‚Äî deliberately *not* the nav wrapper, since the header stays visible above the menu and `menu-btn` itself must stay reachable to close it) ‚Äî keep both lists in sync with whatever body-level siblings exist.
- Elements that are invisible-but-present (`opacity: 0`/`pointer-events: none`, not `display: none`) ‚Äî `#back-to-top` near the top of the page, `#header-logo-link` before scrolling past the hero ‚Äî must have their `tabIndex` toggled between `-1` and `0` alongside their visibility, or they stay keyboard-focusable while invisible with no visible focus indicator. Both now do this correctly; apply the same pattern to any new element using this show/hide technique.
- Any JS that needs to `.focus()` an element immediately after a class change that flips it from `visibility: hidden` to `visible` (e.g. opening the mobile menu) needs to defer that focus call by **two** `requestAnimationFrame` calls, not zero and not one ‚Äî verified empirically: a same-tick or single-rAF-deferred `.focus()` silently no-ops because the CSS-transition-driven visibility change hasn't committed yet. The mobile menu's open handler does this; reuse the pattern rather than rediscovering it.
- `focus:outline-none` on a Tailwind utility class beats the site's global `a:focus-visible, button:focus-visible, input:focus-visible` outline rule by CSS specificity (a class+pseudo-class selector outranks an element+pseudo-class one), regardless of source order. Don't add `focus:outline-none` to a focusable element unless it also supplies its own equally-visible focus style ‚Äî verify with `getComputedStyle(el).outlineColor` after a real focus, not just visually.
